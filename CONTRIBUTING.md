# Contributing to StatForge

We welcome contributions from the community to enhance StatForge. The two primary ways to contribute are by expanding the **Model Plugin Registry** and by adding new **Journal Templates**.

## 1. Expanding the Model Registry

StatForge utilizes a plugin architecture to dynamically load statistical models. All models must inherit from `statforge.models.registry.BaseModel` and utilize the `@register` decorator.

### Creating a New Model Plugin

1.  **Define the Class**: Create a new Python file in `statforge/models/` (or your local `~/.statforge/plugins/` directory for testing).
2.  **Inherit and Register**:
    ```python
    from statforge.models.registry import register, BaseModel

    @register('my_custom_model')
    class MyCustomModel(BaseModel):
        """Docstring describing the model."""

        def fit(self, data, config):
            """
            Implementation for fitting the model.
            Must process the input data and configuration parameters.
            """
            # Implementation here (e.g., using statsmodels or PyMC)
            ...

        def effect_size(self) -> dict:
            """
            Calculates and returns the effect size and its interpretation.
            Returns:
                dict: Must include at least a 'label' key (e.g., 'small', 'medium', 'large').
            """
            return {"my_metric": 0.5, "label": "medium"}
    ```
3.  **Testing**: Ensure you write comprehensive unit tests for your model targeting known datasets to assert statistical correctness.

## 2. Adding Journal Templates

Reports are generated using Jinja2 templates located in `statforge/report/templates/`. You can contribute new templates for specific journal guidelines (e.g., Nature, PLOS ONE, IEEE).

### Creating a New Template

1.  **Define the Template**: Create a new `.j2` file in `statforge/report/templates/` (e.g., `nature.html.j2` or `ieee.tex.j2`).
2.  **Utilize Expected Variables**: The `ReportBuilder` will pass a dictionary of results to your template. Your template should safely access these variables. Common variables include:
    *   `results.methods_text`: The string generated by the `MethodsBuilder`.
    *   `results.tests`: A list of test result dictionaries.
    *   `results.table_rows`: Formatted string arrays for generating tables.
3.  **Example Snippet**:
    ```html
    <!-- nature.html.j2 -->
    <h1>Nature Styled Report</h1>
    <section id="methods">
        <h2>Methods</h2>
        <p>{{ results.get('methods_text', '') }}</p>
    </section>
    <section id="results">
        <h2>Results Table</h2>
        <table>
            <!-- Table implementation iterating over results.table_rows -->
        </table>
    </section>
    ```

## Submitting a Pull Request

1.  Fork the repository and create a feature branch (`git checkout -b feature/your-feature-name`).
2.  Ensure your code passes all linting (`mypy`) and unit tests (`pytest`).
3.  Submit a Pull Request providing a clear description of the new model or template added.
